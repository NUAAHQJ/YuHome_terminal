# 禹家智能家居终端项目介绍

## 项目背景

禹家是一套以 OpenHarmony 大禹开发板为控制中心的智能家居系统。

项目最初采用电脑运行 MQTT Broker 和 HTTP Bridge，大禹负责显示和控制，ESP32 负责采集传感器数据、执行灯光和空调等操作。这种方案适合开发阶段，但不适合长期家庭使用：如果 Broker 必须运行在电脑上，用户就需要在家中长期保持一台电脑开机。

目前项目已经将 Broker 和 HTTP Bridge 迁移到大禹本机。大禹既是交互终端，也是家庭局域网内的通信中心。ESP32 直接连接大禹，日常使用不再依赖电脑。电脑只在编译、安装、串口调试和故障排查时使用。

## 系统组成

系统主要由三部分组成：

1. 大禹终端
2. ESP32 设备控制器
3. 家庭局域网

大禹终端运行“禹家终端”应用，负责界面显示、设备管理、命令路由、身份验证、语音识别、消息加解密以及本地 Broker。

ESP32 根据所连接的硬件声明自身能力。例如，一块 ESP32 可以同时具有温湿度、照明和门锁能力，另一块 ESP32 可以具有空调、红外学习和报警能力。大禹根据能力绑定决定命令应该发送给哪块设备，不要求用户在每次控制前手动切换目标。

局域网负责大禹和 ESP32 之间的无线通信。设备首次接入时通过 USB 串口完成 Wi-Fi、Broker 地址、设备编号和 TLS 参数配置，之后即可脱离 USB 使用。

## 整体架构

```text
                    家庭局域网

  传感器控制器  ───────┐
  照明与门锁           │
                       │   MQTTS / HTTPS
  空调控制器    ───────┼─────────────────  大禹终端
  红外与报警           │                    本地 Broker
                       │                    本地语音识别
  后续新增设备  ───────┘                    人脸与声纹验证
                                            设备管理界面
```

大禹本机开放以下局域网接口：

| 通信方式 | 明文端口 | TLS 端口 |
| --- | ---: | ---: |
| MQTT | 1883 | 8884 |
| HTTP | 3000 | 3444 |

正式使用时，ESP32 默认使用 MQTTS 或 HTTPS。大禹应用内部通过回环地址访问本机 Broker，这部分流量不会离开设备。

## 主要功能

### 设备控制

目前已实现的控制功能包括：

- 照明开关；
- 空调开关、温度和模式控制；
- 窗帘开合度控制；
- 门锁开锁和上锁；
- 安全报警确认与消警。

控制命令不会根据界面当前选中的设备直接发送，而是根据能力绑定路由。例如，用户选中空调控制器时仍然可以开灯或开锁，大禹会分别把命令发送给照明设备和门锁设备。

### 环境监测

传感器设备可以上报：

- 室内温度；
- 室内湿度；
- 烟雾状态；
- 水浸状态；
- 人体感应状态；
- 门锁和照明状态。

室外温湿度和天气由天气服务提供。室内数据与室外数据在总览页分别显示。

### 多设备接入

每块 ESP32 使用 Base MAC 生成唯一设备编号：

```text
esp32-XXXXXXXXXXXX
```

设备上线后发送 `device_hello`，其中包含设备名称、能力、控制通道、加密方式和密钥 epoch。大禹收到后完成设备注册，并自动绑定尚未分配的控制能力。

当前支持在设备绑定界面管理：

- 照明设备；
- 空调设备；
- 窗帘设备；
- 门锁设备。

传感器和报警属于主动上报能力，不需要用户手动选择控制目标。

### 空调万能适配

空调控制器连接红外学习与发射模块，可以学习现有空调遥控器的红外编码。用户按照大禹提示依次学习开机、关机、温度加、温度减、制冷和制热六个按键。

学习结果保存在红外模块自身 Flash 中，正常断电和 ESP32 固件更新不会清除。完成一次学习后，同一型号空调无需重复配置。

### 语音助手

语音模块在大禹本地运行，不依赖云端语音 API。当前流程包括：

1. 持续监听唤醒词“小禹小禹”；
2. 唤醒后进行流式语音识别；
3. 使用本地轻量语义解析判断控制意图；
4. 在需要时进行声纹验证；
5. 向对应设备发送命令；
6. 播放本地结果提示音。

当前支持开灯、关灯、开空调和关空调等语义表达，不要求用户只说固定句式。开门、解锁等高风险语音不会直接执行。

### 人脸与声纹

人脸识别用于门禁授权。用户可以在大禹上录入人脸、查看授权用户、删除用户，并通过识别结果控制门锁。

声纹保护用于语音命令执行前的说话人验证。声纹录入包含三轮采集，可以单独启用或停用授权用户。验证失败时，命令不会发送到设备。

人脸模板和声纹特征均保存在大禹本地，系统不需要上传原始图像或原始录音。

### 安全预警

ESP32 检测到烟雾或水浸异常后，会向大禹上报警情。总览界面显示红色一级预警，并提供明确的“取消报警”按钮。

大禹会记录报警来源设备。用户取消报警时，消警命令发送回实际上报警情的设备，而不是发送给当前选中的设备。收到设备 ACK 后，界面显示预警已经解除。

## 通信与加密

系统支持两种业务通道：

- MQTT；
- HTTP。

两种通道使用相同的设备编号、命令格式和加密帧。切换通道不会改变业务命令含义。

局域网传输可以使用 MQTTS 和 HTTPS。业务消息在 TLS 之上继续使用 SM4-GCM 或 AES-GCM 加密。每个设备分别维护自己的动态密钥、加密算法和 epoch，避免多块设备共用状态导致串包或选错密钥。

动态密钥通过 ECDH 协商产生。设备 ACK 必须带回真实 `deviceId` 和命令 `seq`，大禹同时校验这两个字段后才更新界面状态。

## 界面设计

应用目前保留两个主要页面：

- 总览；
- 设置。

总览页用于日常使用，显示时间、天气、环境数据、门禁、照明、空调、窗帘、语音助手和设备接入入口。

设置页用于维护，包含设备状态、设备接入、空调万能适配、声纹保护、设备绑定、控制通道、加密算法和调试信息。

界面中的设备卡片只显示用户需要理解的状态。模型名称、底层库状态和协议细节集中放在调试信息或设计文档中，不在日常界面中展示。

## 当前完成情况

目前已经完成并实际联调的部分包括：

- 大禹本地 MQTT Broker；
- 大禹本地 HTTP/HTTPS Bridge；
- MQTTS 和 HTTPS；
- MQTT/HTTP 双通道切换；
- SM4-GCM/AES-GCM 切换；
- 多设备注册、独立密钥和能力路由；
- 温湿度、照明、门锁、空调、窗帘和报警；
- USB 串口配网；
- 空调红外学习；
- 本地唤醒、语音识别和语义解析；
- 人脸识别；
- 声纹录入与验证；
- 云端状态上报和本地离线控制。

项目仍处于持续联调阶段。当前重点是长时间运行稳定性、设备断线恢复、TLS 证书部署、界面性能以及更多类型设备的能力接入。
